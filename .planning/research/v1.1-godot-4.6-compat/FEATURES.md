# Feature Research: Godot 4.6/4.6.1 Compatibility Audit

**Domain:** Godot 4.6.x engine changes impacting gdauto CLI tool
**Researched:** 2026-03-28
**Godot versions:** 4.6-stable (Jan 2026), 4.6.1-stable (Feb/Mar 2026)
**Overall confidence:** HIGH (file format changes), MEDIUM (behavioral changes), LOW (undocumented edge cases)

## Critical Format Changes (Must-Fix)

Changes that break gdauto's file generation or parsing if not addressed. Files produced by gdauto v1.0 may fail validation or produce unnecessary diffs when loaded/resaved by Godot 4.6.

### 1. load_steps Removed from Text Resource Headers

| Attribute | Detail |
|-----------|--------|
| **Impact** | CRITICAL -- affects ALL generated files |
| **PR** | godotengine/godot#103352 |
| **Docs issue** | godotengine/godot-docs#11707, godotengine/godot-proposals#11802 |
| **What changed** | Godot 4.6 no longer writes `load_steps=N` in `[gd_scene]` and `[gd_resource]` headers. The attribute was only used for progress bar display during loading and was never functionally required. |
| **Backward compat** | Godot 4.6 still READS files containing `load_steps` without error. However, when Godot 4.6 re-saves a file, it STRIPS `load_steps` from the header. |
| **gdauto impact** | gdauto v1.0 computes and writes `load_steps` in every generated .tres and .tscn file. This means: (a) files generated by gdauto will produce unnecessary diffs when resaved by Godot 4.6, (b) golden files containing `load_steps` will not match Godot 4.6 output, (c) the `sprite validate` command checks `load_steps` correctness, which is now irrelevant for 4.6. |
| **Confidence** | HIGH -- verified via PR, proposal, docs issue, and migration guide |

**Affected code paths (18 references across 8 source files):**

| File | What it does with load_steps |
|------|------------------------------|
| `src/gdauto/formats/tres.py` | `GdResource.load_steps` field, parsed from header, written in `_build_tres_from_model()` |
| `src/gdauto/formats/tscn.py` | `GdScene.load_steps` field, parsed from header, written in `_build_tscn_from_model()` |
| `src/gdauto/sprite/spriteframes.py` | Computes `load_steps = 1 + len(all_subs) + 1` |
| `src/gdauto/sprite/splitter.py` | Computes `load_steps = 1 + len(sub_resources) + 1` (two call sites) |
| `src/gdauto/sprite/atlas.py` | Computes `load_steps = 1 + len(sub_resources) + 1` |
| `src/gdauto/sprite/validator.py` | `_check_load_steps()` validates correctness of load_steps value |
| `src/gdauto/scene/builder.py` | Computes `load_steps = len(ext_resources) + 1` |
| `src/gdauto/tileset/builder.py` | Hardcoded `load_steps=3` |
| `src/gdauto/commands/sprite.py` | Computes `load_steps = 1 + len(all_sub_resources) + 1` |

**Recommendation: MUST-FIX**

Strategy: version-aware generation.
- When targeting Godot 4.6+: omit `load_steps` from generated headers.
- When targeting Godot 4.5: continue including `load_steps` (backward compat).
- Parser: continue accepting `load_steps` when reading (it is harmless and files from 4.5 will have it).
- Validator: skip `_check_load_steps()` when targeting 4.6+, or demote from warning to info.
- Golden files: need 4.6-variant golden files without `load_steps`.
- The `GdResource.load_steps` and `GdScene.load_steps` fields should remain `Optional[int]` (already are), and builders should set them to `None` for 4.6+ output.

### 2. Unique Node IDs (unique_id) Added to .tscn Nodes

| Attribute | Detail |
|-----------|--------|
| **Impact** | MODERATE -- affects scene parsing and generation |
| **PR** | godotengine/godot#106837 |
| **What changed** | Nodes in .tscn files now have an optional `unique_id=NNNNNNNNNN` integer attribute in their header. This is a 32-bit integer that acts as a fallback identifier when nodes are renamed or reparented in inherited/instantiated scenes. |
| **Example** | `[node name="PlayerCamera" type="Camera3D" parent="Player/Head" unique_id=1697057368]` |
| **Backward compat** | Scenes with `unique_id` attributes load in older Godot versions (they ignore unknown attributes). Scenes without `unique_id` load fine in 4.6. |
| **gdauto impact** | (a) The parser in `common.py` already handles arbitrary header attributes via regex, so parsing works. (b) The `SceneNode` dataclass does not store `unique_id`, so it is lost on round-trip when building from model. (c) `_build_tscn_from_model()` in `tscn.py` does not emit `unique_id`. (d) Round-trip via raw sections preserves it (the `_format_header` uses `raw_line`). |
| **Confidence** | HIGH -- verified via PR, docs RST, and release notes |

**Recommendation: SHOULD-SUPPORT**

Strategy: parse and preserve, optionally generate.
- Add `unique_id: int | None = None` to `SceneNode` dataclass.
- Extract from `section.header.attrs.get("unique_id")` in `_extract_node()`.
- Include in `_build_tscn_from_model()` when present.
- For `scene create`: optionally generate unique_id values (Godot uses a random int32; we could use the same algorithm).
- Round-trip via raw sections already works; this only matters for model-based serialization.
- Low risk: omitting `unique_id` in generated scenes is fine; Godot assigns them when the scene is first opened.

## Versioning Strategy Recommendation

gdauto needs a target engine version concept. Recommended approach:

1. **Global setting**: `--godot-version` CLI flag or environment variable (e.g., `GDAUTO_GODOT_VERSION=4.6`)
2. **Auto-detection**: When a Godot binary is available, query its version via `godot --version` and use that
3. **Default**: Target 4.6 (current stable) but generate files that 4.5 can also read
4. **Practical minimum**: Since `load_steps` is the only breaking change and Godot 4.6 still reads files with it, the simplest approach is:
   - For generation: stop emitting `load_steps` (4.6 style). Files are still valid in 4.5 (4.5 ignores missing load_steps; it recomputes on load).
   - For parsing: continue accepting `load_steps` if present (backward compat).
   - This means we can just drop `load_steps` from all builders without version branching.

**Verification needed:** Confirm that Godot 4.5 loads .tres/.tscn files without `load_steps` correctly. If so, the simplest path is to just stop emitting it entirely, making all output forward-compatible.

## Anti-Features (Do NOT Build)

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Version-branched golden files per Godot version | Maintenance burden explodes; two sets of 648 tests | Use version-agnostic comparison (strip/ignore load_steps in comparisons) or just target 4.6 format |
| Automatic scene migration (4.5 -> 4.6 format) | Godot already does this on first open; duplicating engine logic is fragile | Document that users should open projects in Godot 4.6 to upgrade |
| TileMapLayer scene data manipulation | Complex, scene-specific, editor has good tools | Keep scope to TileSet .tres generation |
| GDExtension tooling | Different domain entirely; gdauto is about file formats and headless workflows | Out of scope |
| Editor plugin compatibility layer | gdauto is CLI, not editor plugin | Out of scope |

## MVP Recommendation for v1.1

Priority order based on impact and effort:

1. **Drop load_steps from all builders** (MUST-FIX, LOW effort, HIGH impact)
   - Verify 4.5 reads files without load_steps
   - Remove load_steps computation from all 6 builder files
   - Set load_steps=None in all GdResource/GdScene construction
   - Update _build_tres_from_model and _build_tscn_from_model (already handle None)

2. **Update validator to skip load_steps check** (MUST-FIX, LOW effort)
   - Make _check_load_steps a no-op or remove it
   - Alternatively: only warn if load_steps is present AND wrong

3. **Update golden files** (MUST-FIX, LOW effort)
   - Remove load_steps from golden file headers
   - Or: make golden comparison strip load_steps before diffing

4. **Add unique_id to SceneNode** (SHOULD-SUPPORT, LOW effort)
   - Add field, extract in parser, emit in serializer
   - Preserves round-trip fidelity for 4.6 scenes

5. **Run E2E tests against Godot 4.6.1** (SHOULD-TEST, LOW effort)
   - Install 4.6.1, run full test suite
   - Document any failures

6. **Update project create defaults** (NICE-TO-HAVE, LOW effort)
   - Add Jolt physics setting for 4.6+ targets
   - Cosmetic, not blocking
