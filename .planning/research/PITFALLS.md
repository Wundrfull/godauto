# Domain Pitfalls: Godot 4.5 to 4.6.1 Migration

**Domain:** Godot CLI tooling, multi-version .tscn/.tres generation, headless pipeline
**Researched:** 2026-03-28
**Scope:** Pitfalls when updating gdauto from Godot 4.5-only to 4.5+4.6.x dual support
**Confidence:** HIGH (all critical pitfalls verified through official PRs, Godot docs, and GitHub issues)

---

## Critical Pitfalls

Mistakes that cause broken output, silent data corruption, or test suite instability.

---

### Pitfall 1: load_steps Removal Breaks Generated .tscn Files in Both Directions

**What goes wrong:**
Godot 4.6 no longer writes `load_steps` to .tscn or .tres files (PR #103352, merged Dec 2025). gdauto currently generates `load_steps` in both `_build_tscn_from_model()` (`src/gdauto/formats/tscn.py:221-222`) and `_build_tres_from_model()` (`src/gdauto/formats/tres.py:188-189`). This creates two problems:

1. **Forward problem:** Files generated by gdauto with `load_steps` will work in Godot 4.6 (it still parses the attribute), but when Godot 4.6 re-saves them, it strips `load_steps`, creating diff noise in version control. Users will see phantom changes in every generated file after opening it once in 4.6.

2. **Backward problem:** If gdauto stops emitting `load_steps` to match 4.6 behavior, the generated files still load fine in Godot 4.5 (it tolerates missing `load_steps`). However, files without `load_steps` will diff against 4.5-generated golden files.

**Why it happens:**
`load_steps` was never used internally by the engine; it only served progress bar hints for loading. Its removal was not announced in the 4.6 release notes (only documented after community report in godot-docs#11707).

**Our specific exposure:**
- `src/gdauto/formats/tscn.py` line 221: `_build_tscn_from_model()` emits `load_steps` when `scene.load_steps is not None`
- `src/gdauto/formats/tres.py` line 189: `_build_tres_from_model()` emits `load_steps` when `resource.load_steps is not None`
- `src/gdauto/scene/builder.py` line 37: `build_scene()` computes `load_steps = len(ext_resources) + 1`
- All golden files in `tests/fixtures/golden/` include `load_steps` in their expected output
- `tests/unit/test_golden_files.py` compares against committed references containing `load_steps`

**Prevention:**
- Add a `godot_version` parameter (or detect from project) to control output format
- For 4.6+: omit `load_steps` from generated headers entirely
- For 4.5: continue emitting `load_steps` for clean round-trips
- Default behavior: omit `load_steps` (4.6 is the current stable; 4.5 tolerates omission)
- Update all golden files to omit `load_steps`
- Parser must continue accepting `load_steps` when present (backward compatibility)

**Detection:**
- Golden file tests fail after updating expected outputs
- Users report "noisy diffs" when opening gdauto-generated files in Godot 4.6

**Severity:** HIGH (affects every generated .tscn and .tres with external dependencies)

**Phase to address:** First phase of v1.1 (format compatibility layer)

### Pitfall 2: Missing unique_id on Generated [node] Sections Causes Export Non-Determinism

**What goes wrong:**
Godot 4.6 added `unique_id` attributes to every `[node]` section in .tscn files (PR #106837). These are 32-bit integers stored as unquoted numeric attributes. When a .tscn file without `unique_id` values is opened in Godot 4.6, the engine assigns IDs non-deterministically at runtime. This causes:

1. Every export produces different binary hashes (issue #115971, "Very Bad" severity)
2. Patch PCK exports include unchanged scenes because IDs regenerate each time
3. Re-saving a gdauto-generated scene in 4.6 adds `unique_id` to every node, creating massive diffs

Example of the new format:
```
[node name="PlayerCamera" type="Camera" parent="Player/Head" unique_id=1697057368]
```

**Why it happens:**
The `unique_id` system was designed for editor-side scene inheritance robustness. External tools that generate .tscn files do not know about this system and produce files without IDs. Godot 4.6 assigns IDs on load, but the assignment is not deterministic (issue #112332 documents seed problems in `generate_scene_unique_id()`).

**Our specific exposure:**
- `src/gdauto/formats/tscn.py` line 246-253: `_build_tscn_from_model()` emits `[node name=... type=... parent=...]` with no `unique_id`
- `src/gdauto/scene/builder.py`: `build_scene()` creates `SceneNode` instances with no `unique_id` field
- The `SceneNode` dataclass (tscn.py line 35-42) has no `unique_id` field at all
- Round-trip parsing also does not extract `unique_id` from node headers

**Prevention:**
- Add `unique_id: int | None = None` field to `SceneNode` dataclass
- Update parser (`_extract_node`) to read `unique_id` from section header attrs
- Update serializer (`_build_tscn_from_model`) to emit `unique_id=N` when present
- For 4.6+ generation: assign `unique_id` values using a CSPRNG (32-bit random int)
- For 4.5 generation: omit `unique_id` (4.5 ignores unknown attributes gracefully)
- Default behavior: emit `unique_id` (4.5 tolerates it; 4.6 needs it)
- Normalize `unique_id` in golden file comparisons (add pattern to `normalize_for_comparison()`)

**Detection:**
- Godot 4.6 reports non-deterministic exports with gdauto-generated scenes
- Opening a generated .tscn in 4.6, then closing, produces a diff with `unique_id` on every node

**Severity:** HIGH (affects every generated .tscn file; causes real workflow breakage in 4.6)

**Phase to address:** First phase of v1.1 (format compatibility layer)

### Pitfall 3: Golden File Tests Become Fragile Across Engine Versions

**What goes wrong:**
The current golden file infrastructure (`tests/unit/test_golden_files.py`) normalizes UIDs and resource IDs but assumes a single output format. With 4.5 and 4.6 producing different serializations (load_steps present/absent, unique_id present/absent), golden files must either:

1. Target one version (breaking comparisons against the other), or
2. Maintain parallel golden file sets (doubling maintenance burden), or
3. Normalize version-specific differences before comparison

The third option is best but requires careful design to avoid masking real bugs.

**Why it happens:**
The normalization function (`normalize_for_comparison()`) was designed for random values (UIDs, resource IDs) that differ between runs. Version-specific structural differences (missing attributes, new attributes) are a different category that simple regex substitution cannot handle cleanly.

**Our specific exposure:**
- `tests/unit/test_golden_files.py`: `normalize_for_comparison()` handles UIDs and resource IDs only
- `tests/fixtures/golden/`: committed reference files contain `load_steps` values
- `test_golden_spriteframes()`, `test_golden_tileset()`, `test_golden_scene()`: compare exact normalized output

**Prevention:**
- Add `load_steps` stripping to `normalize_for_comparison()`: remove entire `load_steps=N` attribute from header lines
- Add `unique_id` stripping to `normalize_for_comparison()`: remove `unique_id=N` from node header lines
- Update golden reference files to the "canonical" format (4.6 style: no load_steps, no unique_id in golden files since they are normalized away)
- Consider a version-parameterized fixture that generates with `--target-version 4.5` and `--target-version 4.6` and validates both
- For E2E tests: detect the installed Godot version and choose appropriate validation criteria

**Detection:**
- Golden file tests fail immediately when switching gdauto's output format
- Tests pass locally but fail in CI if CI uses a different Godot version

**Severity:** HIGH (test suite becomes unreliable, masking real regressions)

**Phase to address:** First phase of v1.1 (alongside format changes)

### Pitfall 4: load_steps Removal Also Affects .tres Files, Not Just .tscn

**What goes wrong:**
PR #103352 modified `resource_format_text.cpp`, which handles both .tscn and .tres serialization. The `load_steps` removal applies to both file types. Our v1.0 PITFALLS.md (Pitfall 8) documented format instability between minor versions but only anticipated .tscn changes. SpriteFrames .tres files and TileSet .tres files both currently include `load_steps` in their headers.

**Why it happens:**
The .tres and .tscn formats share the same serializer in Godot's codebase. Changes to the serializer affect both file types.

**Our specific exposure:**
- `src/gdauto/sprite/spriteframes.py`: `build_spriteframes()` creates `GdResource` with computed `load_steps`
- `src/gdauto/tileset/builder.py`: `build_tileset()` creates `GdResource` with computed `load_steps`
- Golden files for SpriteFrames and TileSet both include `load_steps`

**Prevention:**
- Apply the same `load_steps` omission logic to .tres generation as to .tscn
- Update `GdResource` builder functions to optionally omit `load_steps`
- When `load_steps` is None, the serializer already omits it; so set `load_steps=None` in builders for 4.6+ mode
- Test: generate a SpriteFrames .tres, open in Godot 4.6, save, verify no diff

**Detection:**
- SpriteFrames golden file test fails when expected output is updated to omit `load_steps`
- TileSet golden file test shows same symptom

**Severity:** HIGH (every generated .tres file is affected, not just scenes)

**Phase to address:** First phase of v1.1 (format compatibility layer)

### Pitfall 5: Headless Import Race Condition Persists in 4.6.x

**What goes wrong:**
The `--headless --import --quit` race condition (issue #77508) remains unfixed as of Godot 4.6.1. The import scan thread can exit before completing when `--quit` or `--quit-after 1` is used. Our backend uses `--quit-after 30` (line 161 of `backend.py`), which is safe, but:

1. No verification that import actually completed (we do not check `.godot/imported/` contents)
2. The retry logic in `export/pipeline.py` retries on GdautoError (non-zero exit code) but the import race condition often succeeds with exit code 0 despite incomplete imports

**Why it happens:**
Godot's import uses `call_deferred()` which requires frame ticks. `--quit` exits before deferred calls run. The 4.6 release did not include a fix for this long-standing issue.

**Our specific exposure:**
- `src/gdauto/backend.py` line 161: `import_resources()` uses `--quit-after 30`
- `src/gdauto/export/pipeline.py` line 24: `check_import_cache()` only checks directory existence, not completeness
- E2E tests may pass intermittently if import partially completes

**Prevention:**
- After `import_resources()` returns, verify that expected files exist in `.godot/imported/`
- Add a `verify_import` method that checks for specific resource types the project uses
- For E2E tests: use a session-scoped fixture that imports once and verifies completeness
- Keep `--quit-after 30` (not `--quit`); this is already correct in our code
- Log the Godot version on first use so users can correlate issues with engine versions

**Detection:**
- E2E tests fail with "Failed loading resource" after what appeared to be a successful import
- Export commands produce incomplete PCK files

**Severity:** MODERATE (our existing `--quit-after 30` mitigates this; the issue is verification, not the core mechanism)

**Phase to address:** Second phase of v1.1 (headless pipeline hardening)

### Pitfall 6: AnimationPlayer String-to-StringName Type Change Affects Inspection Output

**What goes wrong:**
Godot 4.6 changed AnimationPlayer properties from `String` to `StringName` type (migration guide confirms: `assigned_animation`, `autoplay`, `current_animation` all changed from `String` to `StringName`). This means .tscn files saved in 4.6 will serialize these as `&"animation_name"` (StringName syntax) instead of `"animation_name"` (String syntax). Our parser handles both forms, but the `resource inspect` and `scene list` JSON output will show different types for the same property depending on which Godot version saved the file.

**Why it happens:**
StringName and String have different serialization in .tscn format. The 4.6 migration changed the declared property types, so the engine now serializes them with the `&"..."` syntax.

**Our specific exposure:**
- `src/gdauto/formats/values.py` line 679-681: `parse_value()` correctly handles `&"..."` syntax (returns `StringName` dataclass)
- `src/gdauto/formats/values.py` line 873-876: `GodotJSONEncoder` serializes `StringName` as its Godot text representation
- `scene list` output will show `&"idle"` for 4.6-saved files vs `"idle"` for 4.5-saved files

**Prevention:**
- This is informational; no fix needed in the parser
- Document in `resource inspect` and `scene list` help text that output reflects the file's serialized types
- If consistent output across versions is desired, normalize StringName to String in JSON output (consider this a UX choice, not a bug)
- Do NOT force all strings to StringName or vice versa; the type information is meaningful

**Detection:**
- Users report inconsistent JSON output for the same scene opened in different Godot versions
- Tests comparing `scene list` output fail when switching engine versions

**Severity:** MODERATE (cosmetic difference in inspect output; no functional impact)

**Phase to address:** Third phase of v1.1 (inspection command updates)

### Pitfall 7: AnimationLibrary Serialization Change Breaks Round-Trip for Animation Resources

**What goes wrong:**
Godot 4.6 changed AnimationLibrary serialization to "avoid using Dictionary" (GH-110502, mentioned in 4.6 beta 1 notes). AnimationLibrary .tres files saved in 4.6 use a different property structure than 4.5. If gdauto parses a 4.5-style AnimationLibrary and a user opens it in 4.6, Godot silently upgrades the format. Re-saving produces a different structure, breaking round-trip fidelity.

**Why it happens:**
The internal serialization format for AnimationLibrary changed from a Dictionary-based format to a sub-resource-based format. This is a resource-type-specific change within the same `format=3` container.

**Our specific exposure:**
- `src/gdauto/formats/common.py`: The parser handles dictionaries and sub-resources generically, so parsing is not affected
- gdauto does not generate AnimationLibrary resources directly (SpriteFrames is a separate resource type)
- `resource inspect` will show different structures for the same AnimationLibrary depending on version
- Round-trip tests that parse a 4.5 AnimationLibrary and serialize it back will produce 4.5-format output; if validated against 4.6, the diff will be significant

**Prevention:**
- gdauto's SpriteFrames resources are NOT AnimationLibrary resources; they are SpriteFrames (separate class). Verify that SpriteFrames serialization is unaffected by this change.
- For `resource inspect`: accept both formats gracefully (our generic parser already does this)
- Add a test that parses a 4.6-saved AnimationLibrary .tres to verify our parser handles the new format
- Do NOT try to convert between formats; let Godot handle its own format upgrades

**Detection:**
- `resource inspect` on a 4.6-format AnimationLibrary shows unexpected structure
- Round-trip test writes a .tres that Godot 4.6 then upgrades on load

**Severity:** MODERATE (gdauto does not generate AnimationLibrary files; only affects inspect/parse of existing ones)

**Phase to address:** Second phase of v1.1 (parser validation against 4.6 files)

### Pitfall 8: config_version Remains 5 but Parsing Assumptions May Still Break

**What goes wrong:**
Godot 4.6 keeps `config_version=5` in project.godot (verified: `CONFIG_VERSION = 5` in `core/config/project_settings.h` on master). However, new default project settings were added (D3D12 as Windows default, Jolt Physics as 3D default, changed glow defaults), and the `project validate` command may flag these as issues when validating a 4.6 project with 4.5 assumptions.

**Why it happens:**
`config_version` only bumps on structural format changes, not on new setting additions or default value changes. The version did not change from 4.0 through 4.6.

**Our specific exposure:**
- `src/gdauto/formats/project_cfg.py`: Custom parser that reads INI-like format
- `src/gdauto/commands/project.py`: `project validate` and `project info` commands
- If gdauto validates project settings against hardcoded "expected" values, 4.6 defaults will differ

**Prevention:**
- Do NOT check `config_version` to determine engine version (it is always 5 for all 4.x)
- To detect engine version from project, check `[application] config/features` which lists the engine version
- When validating projects, do not flag unknown settings as errors; new Godot versions add settings
- Store the detected Godot version (from binary `--version` or from project features) and use it to adjust validation rules

**Detection:**
- `project validate` reports false positives on 4.6 projects
- `project info` shows config_version=5 for both 4.5 and 4.6, giving no version information

**Severity:** MODERATE (false validation errors are confusing but not destructive)

**Phase to address:** First phase of v1.1 (version detection)

### Pitfall 9: Export Determinism Regression for Scenes Without unique_id

**What goes wrong:**
Godot 4.6 has a known regression (issue #115971, marked "Very Bad", milestoned for 4.7) where exporting scenes that lack `unique_id` values produces non-deterministic output. The `unique_id` is regenerated on every export because the RNG seed is not stable. This means gdauto's `export release/debug/pack` commands will produce different output each time if the project contains scenes generated by gdauto (which do not have `unique_id`).

**Why it happens:**
PR #106837 added unique_id to every node but the ID generation is not deterministic for scenes that have not been re-saved in 4.6. The RNG seed gets overwritten by editor configuration saves (issue #112332).

**Our specific exposure:**
- `src/gdauto/commands/export.py` and `src/gdauto/export/pipeline.py`: Export commands invoke Godot's export pipeline
- gdauto-generated .tscn files have no `unique_id` values
- Users who generate scenes with gdauto then export with gdauto will see non-deterministic exports

**Prevention:**
- Generate `unique_id` on all nodes when targeting 4.6+ (see Pitfall 2)
- Document this as a known Godot 4.6 issue in gdauto's help/docs
- Advise users to "Upgrade Project Files" in Godot editor before using gdauto's export commands
- Consider adding a `gdauto project upgrade-scenes` command that adds `unique_id` to all .tscn files

**Detection:**
- Two successive `export pack` commands produce PCKs with different hashes
- Patch PCK exports include scenes that have not actually changed

**Severity:** MODERATE (real workflow impact but caused by upstream Godot bug, not gdauto)

**Phase to address:** First phase of v1.1 (format compatibility; generate unique_id)

### Pitfall 10: Version Detection Strategy Determines Entire Compatibility Architecture

**What goes wrong:**
Adding multi-version support requires a strategy for determining which Godot version to target when generating files. There are multiple sources of version information, and choosing the wrong strategy leads to either over-complex code or incorrect output:

- `GODOT_PATH` binary `--version` output: accurate but requires binary on PATH
- `project.godot` `[application] config/features`: lists engine version but may be stale
- `config_version`: always 5 for all 4.x; useless for minor version detection
- CLI flag `--target-version`: explicit but burdensome for users
- Generated file contents: cannot determine version from format=3 alone

**Prevention:**
- Primary: detect from Godot binary if available (`backend._check_version()` already parses this)
- Secondary: read `config/features` from project.godot
- Override: `--target-version` CLI flag
- Default: 4.6+ behavior (current stable; produces files that work in both 4.5 and 4.6 for pure-Python commands)
- Store detected version in a context object accessible to all commands

**Severity:** LOW (architectural decision, not a runtime failure)

**Phase to address:** First phase of v1.1 (design the version detection strategy before implementing format changes)

### Pitfall 11: Normalization Regex Drift Between Parser and Test Infrastructure

**What goes wrong:**
Adding `unique_id` normalization to `normalize_for_comparison()` requires a new regex pattern. If the pattern is too aggressive (e.g., matching any `unique_id=N` anywhere in the file), it could mask legitimate test failures. If too narrow, it misses variations in formatting.

**Our specific exposure:**
- `tests/unit/test_golden_files.py` lines 39-43: five regex patterns for normalization
- New pattern needed: `unique_id=\d+` in node header lines
- Risk: `unique_id` could appear as a property name in a user's custom resource; normalizing it everywhere would hide real differences

**Prevention:**
- Use an anchored pattern that only matches `unique_id=\d+` within `[node ...]` brackets
- Pattern: `r'\[node ([^]]*?)unique_id=\d+([^]]*?)\]'` with replacement that strips the attribute
- Add unit tests for the normalization pattern itself (like the existing `test_normalize_*` tests)
- Never use a global `unique_id=\d+` replacement outside of node headers

**Severity:** LOW (test infrastructure detail, but incorrect normalization can hide bugs)

**Phase to address:** First phase of v1.1 (test infrastructure updates)

### Pitfall 12: Godot 4.6.1 Animation Event Loss on Version Mismatch

**What goes wrong:**
Community reports indicate that animation events manually created in 4.6.0 can disappear when opening projects in 4.6.1 (issue #116408). While gdauto does not directly create animation events (it creates SpriteFrames, not AnimationPlayer resources with method tracks), users who combine gdauto-generated SpriteFrames with manually-created AnimationPlayer resources could lose their animation events if they upgrade between 4.6.0 and 4.6.1.

**Our specific exposure:**
- gdauto does not generate AnimationPlayer resources or animation events
- However, `resource inspect` should be able to read AnimationPlayer .tres files from both 4.6.0 and 4.6.1 without data loss

**Prevention:**
- No code changes needed for gdauto
- Document this as a known Godot issue in gdauto's compatibility notes
- If adding AnimationPlayer support in the future, test against 4.6.1+ (not 4.6.0)

**Severity:** LOW (does not affect gdauto's generated resources; informational only)

**Phase to address:** Documentation only (v1.1 release notes)

### Pitfall 13: Quaternion Initialization Change May Affect Scene Property Defaults

**What goes wrong:**
Godot 4.6 changed Quaternion default initialization from zero `Quaternion(0, 0, 0, 0)` to identity `Quaternion(0, 0, 0, 1)`. This means .tscn files saved in 4.6 will not include `rotation` properties for nodes at identity rotation (because the default matches), while 4.5 files might include explicit `rotation = Quaternion(0, 0, 0, 0)`. This is a 3D concern and gdauto primarily generates 2D resources, but `scene create` and `resource inspect` should handle both cases.

**Prevention:**
- Our parser already handles both forms (it parses whatever values are present)
- No code changes needed; just be aware that 4.5 and 4.6 .tscn files may have different default-elision behavior
- If a round-trip test fails because a property that was explicit in 4.5 is now default-elided in 4.6, that is expected behavior, not a bug

**Severity:** LOW (primarily affects 3D; gdauto is 2D-focused)

**Phase to address:** No code change needed; awareness only
